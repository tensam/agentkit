# FRAMEWORK.md — 团队工程宪法（跨项目复用）

> 本文件定义所有项目必须遵守的工程纪律。不包含任何项目特定内容。

---

## 1. Test-Driven Development (TDD)

- 任何行为变更或新功能：**先写/更新测试**
- 偏好小而聚焦的测试：每个测试一个断言主题
- 代码 "完成" 的定义 = 测试通过（unit + integration）
- 修改已有测试必须说明原因（什么契约变了）
- 涉及外部接口行为变更，必须有集成测试 + 真实 payload 验证

---

## 2. 可复用性 & 设计

- 偏好小纯函数 + dataclass，避免大方法
- 禁止跨层 copy-paste；共享逻辑抽取到单一 helper/module
- 公共接口必须稳定 & typed；不泄漏内部实现细节
- 仅在行为非显而易见时加最小 docstring

---

## 3. I/O & 异步纪律

- 所有网络 I/O 必须走客户端抽象层（禁止在业务层裸调 HTTP）
- timeout / retry / rate-limit 集中管理
- 异步路径中禁止同步阻塞调用

---

## 4. 变更控制

- 未经明确指示，禁止改变核心业务语义
- 不确定行为是否正确时：先加集成测试用真实数据验证
- 每次变更必须输出：
  1. 增/改了什么测试
  2. 改了哪些文件 & 为什么
  3. 如何运行相关测试

---

## 5. Definition of Done (DoD)

一项变更完成的条件：

- [ ] 测试先行（diff 中可见 TDD 证据）
- [ ] 所有相关测试本地通过
- [ ] 未引入层违规
- [ ] 共享逻辑已抽取可复用抽象
- [ ] memory/STATE.md 和 memory/LOG.md 已更新

---

## 6. 研究优先原则

遇到不明确的外部接口 / 业务逻辑时：

1. 先查项目 KNOWLEDGE 中的已知经验
2. 搜索官方文档 / 作者原始说明
3. 从真实数据（截图 / 抓包 / API 响应）中提取精确参数
4. 利用定义类 API 反查完整配置

**禁止**：未穷尽研究手段就猜测或替代。

---

## 7. 分层纪律

- 每层只依赖下层，禁止反向依赖
- 配置集中管理（config dataclass），不硬编码
- 层间通过明确接口通信，不共享内部状态

---

## 8. 经验沉淀规则

发现任何"以后会再用到的技巧 / 坑 / 链接"时：

1. 写入 `memory/KNOWLEDGE/` 对应文件
2. 在 `INDEX.md` 增加关键词索引
3. 在 `LOG.md` 追加记录
